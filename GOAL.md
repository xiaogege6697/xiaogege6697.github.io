# 项目目标

让第一次访问的人在 10 秒内理解：**xiaogege6697 是一个把复杂 AI 系统做得可执行、可沉淀、可验收的独立构筑者。**

## 成功标准

1. 首屏建立清晰的个人 IP，而不是普通 GitHub 仓库列表。
2. 项目以能力体系呈现，访问者能理解项目之间的关系。
3. 交互帮助理解内容，而非只做装饰。
4. 桌面端与移动端均可读、可操作，并能用证据说明新版本改善了什么。

## 窄腰边界

- GitHub 当前公开仓库是项目数据的唯一事实来源；不保留已删除仓库，不编造个人经历或项目指标。
- 保持纯静态、零后端、可直接部署到 GitHub Pages。
- 保持三个核心 DNA：智能系统构筑者的身份、可执行/可沉淀/可验收的认知、浅色编辑式系统工作台的体验；避免退化成通用 AI Dashboard。
- 不固定具体实现；允许有证据支持的设计升级，只要更接近目标且验收无退化。

## 升级方法

每轮只选择一个主要变化，开始前写清：本轮目标、主要变量、保持不变的核心 DNA。其他修改仅服务于该目标，避免同时改变字体、色彩、布局、文案和动效后无法判断效果来源。

完成后用“改善 / 持平 / 退化”比较新旧版本，每项必须附截图、测试或真实页面证据：

| 维度 | 判断问题 | 结果与证据 |
|---|---|---|
| 身份识别 | 访问者能否在 10 秒内知道我是谁？ | 改善 / 持平 / 退化 + 证据 |
| 独特性 | 换掉名字后，网站是否仍有自身辨识度？ | 改善 / 持平 / 退化 + 证据 |
| 理解效率 | 能否理解项目之间的能力关系？ | 改善 / 持平 / 退化 + 证据 |
| 视觉一致 | 新内容是否像从现有系统自然生长出来？ | 改善 / 持平 / 退化 + 证据 |
| 工程质量 | 移动端、交互、性能和可访问性是否正常？ | 改善 / 持平 / 退化 + 证据 |

发布条件：身份识别与工程质量不得退化，至少一个维度有明确改善；只有主观感觉、没有证据，不发布。基准图是比较对象，不是禁止进步的模板。

## 验收证据

- `npm run validate` 通过。
- 实际打开网页检查桌面端与移动端。
- 验证项目数量、筛选、搜索、排序和详情弹窗。
- 与 `docs/baselines/desktop-v1.png` 做前后对比并完成五项判断。
- 发布后确认线上地址返回成功，并以 GitHub 最新公开仓库复核项目数量。
